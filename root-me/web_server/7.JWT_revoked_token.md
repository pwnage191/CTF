#### Challenge Name
- JWT - Revoked token

#### Vulnerability Type
- JWT authentication bypass

#### Goal
- Access the `/admin` endpoint to retrieve the flag, despite the token being immediately blacklisted after login.

#### Initial Observation
#### Source Code analysis
- The login function generates a JWT for the user and adds that token to a blacklist before sending it back
- The admin endpoint reads the token from the request and checks whether that exact token exists in the blacklist to decide access
- A cleanup function checks blacklisted tokens, and if a token is expired or invalid (cannot be decoded), it removes it from the blacklist

#### Libraries:
- `flask_jwt_extended`: Used in endpoints with @jwt_required to validate the JWT token structure, signature, and expiry during
  incoming requests
- `PyJWT` (`import jwt`): Used in the cleanup function to decode and verify tokens
- Behavior difference
    - `PyJWT` is strict and throws errors for invalid or expired tokens
    - `flask_jwt_extended` may handle validation differently
- Resulting conflict: Same token can be treated differently by each library, leading to inconsistent validation behavior

#### Exploitation
Endpoints discovered:
    - POST : /web-serveur/ch63/login  
	- GET : /web-serveur/ch63/admin
- Source code is given in the challenge.
- Initial tests:  
	- POST /login without Content-Type = 400 Bad Request
	- GET /admin without token = 401 Missing Authorization Header 
	- POST /admin = 405 Method Not Allowed 
	- OPTIONS /admin = Allow: HEAD, GET, OPTIONS
- I added the content type in the POST/login with username and password given in the source code.
- I got the token.
- I tested, whether the algorithm is mandatory or not, alg:none attack but it displayed "alg value not allowed"
- To confirmed that token was revoked, I used it in `Authorization header ` at GET /admin
- It displayed "Token was revoked"
- Modified the token slightly (changed one character in signature)
- Sent modified token in request
- Access was granted and flag retrieved
- Reason:
    - Modified token becomes a different string
    - It is no longer found in the blacklist
    - Server accepts it and grants access

#### Remediation
- Use JTI instead of full token for blacklisting
- Check JTI on every request
- Avoid multiple JWT libraries
- Enforce strict signature validation

#### What I learned
- Multiple libraries can cause inconsistent validation
- Token string is not the same as token identity
- Blacklisting full token strings is insecure
