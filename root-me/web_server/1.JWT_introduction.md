#### Challenge Name  
JWT Introduction

#### Goal  
Connect to the application as admin.

#### Initial Observation
- Logged in as a guest user
- Intercepted the request using Burp Suite
- Observed that the application uses a JWT stored in a browser cookie
- The token contained three sections separated by dots    

Header: {"typ":"JWT","alg":"HS256"}  
Payload: {"username":"guest"}  
Signature: OnuZnYMdetcg7AWGV6WURn8CFSfas6AQej4V9M13nsk

- Decoding the token showed that the `username` field controls access privileges
- Attempted to change `username` to `admin` and modify the algorithm to `none`
- Normally, when `alg` is `HS256`, the server verifies the token by computing the signature using `header.payload` and a secret key 
- If the algorithm is set to `none`, the server may skip signature verification and only decode the token
- In this case, the token can be sent without a signature
    

#### Approach
- Modified the header to set `alg` to `none`
- Modified the payload to change `username` from `guest` to `admin`
- Removed the signature section 
- The application accepted the token even without the trailing dot (header.payload), indicating loose JWT parsing and skipped signature          verification
- Example structure: base64(header).base64(payload)

#### Solution
- Replaced the original cookie value with the forged JWT
- The server accepted the token and authenticated the session as the admin user
    
#### What I Learned
- If a server accepts `alg: none`, signature verification can be bypassed  
- Attackers can modify the payload without knowing the secret key
- Secure implementations must explicitly reject the `none` algorithm and enforce signature validation
