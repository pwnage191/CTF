#### Challenge Name  
JWT – Weak Secret Key

#### Vulnerability Type  
- JWT Authentication Bypass – Weak Secret Key. 
- The application signs JWTs using a weak secret key with the HS256 algorithm, allowing the key to be brute-forced and enabling 
attackers to generate valid tokens.

#### Goal  
Access the admin endpoint and retrieve the flag.

#### Initial Observation
- Accessed the endpoint `/hello`
- The response displayed the following message:
  {"message": "Let's play a small game, I bet you cannot access to my super secret admin section. Make a GET request to /token and
  use the token you'll get to try to access /admin with a POST request."}
- Sent a request to the `/token` endpoint
- The endpoint returned a JWT
- Decoded the token and observed that it was signed using `HS256`
- Since `HS256` uses a shared secret key, the security of the token depends on the strength of that key
    
#### Exploitation
- Extracted the JWT returned by the `/token` endpoint
- Attempted to recover the signing key by performing a brute-force attack
- Used Hashcat with a wordlist to crack the JWT secret key
- After running the attack, the secret key was successfully discovered
- Using the recovered key, generated a new JWT
- Modified the payload to obtain admin access {"role":"admin"}
- Sent a POST request to `/admin` with the forged token
- The server responded with:
  {"message": "method to authenticate is: 'Authorization: Bearer YOURTOKEN'"}
- Added the required header: Authorization: Bearer <forged_token>
- Sent the request again with the header included
- The server accepted the token and returned the flag
    

#### What I Learned
- JWT security depends heavily on the strength of the secret key
- If a weak secret is used with `HS256`, it can be brute-forced
- Once the secret key is discovered, attackers can generate valid tokens and escalate privileges
