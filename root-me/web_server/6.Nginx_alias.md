#### Challenge Name
- Nginx – Alias Misconfiguration  

#### Vulnerability Type
- Misconfiguration  
- Path Traversal via Nginx Alias  

#### Goal
- Access restricted files by exploiting improper alias configuration  

#### Initial Observation
- Opened the website -> login page  
- Viewed source and found:
  `<!--TODO: Patch /assets/ -->`

#### Walkthrough
- Visited `/assets/` and saw limited index access, `../` redirected back  
- Tried path traversal using `/assets/../` but it didn’t work because it gets normalized to `/` before alias is applied, 
  so it no longer matches the `/assets/` location  
- Then tried `/assets../` which revealed additional files including `flag.txt`  
- Accessed `/assets../flag.txt` and retrieved the flag
- <img width="600" height="379" alt="image" src="https://github.com/user-attachments/assets/45a2caee-06ae-48cd-951f-d763504cb607" />
 
- Reason:  
    - The application used an Nginx alias mapping `/assets/ -> /var/www/static/`.  
    - Using `/assets../` caused alias to be applied first: `/var/www/static/../flag.txt`  
    - Then `..` moved one level up: `/var/www/flag.txt`  

#### Remediation
- Configure alias properly with strict path handling  
- Prevent usage of `..` in paths  
- Disable directory listing  
- Follow secure Nginx configuration practices  

#### What I learned
- Try different traversal patterns, not just standard ones  
- Alias misconfiguration can allow directory escape  
- Small hints can reveal attack surface
