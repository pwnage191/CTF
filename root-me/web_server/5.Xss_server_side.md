#### Challenge Name  
XSS - Server Side

#### Goal
- Read the contents of /flag.txt from the server by exploiting the certificate generation feature.

#### Initial Observation
- Opened the web application and identified three endpoints: /signup.php, /login.php, /generate.php
- The application allows users to sign up, log in, and generate a PDF certificate with a custom message
- The backend was identified as PHP/8.2.25 through the X-Powered-By response header
- After login, the first name and last name provided during signup were reflected in the certificate
- The generated certificate was returned as a downloadable PDF file
    
#### Approach
- Tested HTML injection in the message input using `<b>Hello</b>`
- The generated PDF displayed the text literally, indicating that HTML was escaped in the message field
- Tested HTML injection in the signup first name field using `<b>challenger</b>`
- The homepage displayed the name as literal text, meaning HTML was escaped in the web page
- Generated the certificate and observed that the name appeared bold in the PDF
- This indicated that the PDF renderer processed raw HTML and did not escape the name fields
- Confirmed that the injection point existed in the name fields used by the PDF rendering engine
- Injected `<iframe src="file:///flag.txt" width="800" height="400"></iframe>` as the first name during signup
- Logged in and generated the certificate
- The PDF renderer loaded the contents of /flag.txt inside the iframe

#### Solution
- Signed up using the iframe payload in the first name field
- Logged in to the application
- Generated the certificate through /generate.php
- The PDF displayed the contents of /flag.txt

#### What I Learned
- Learned that server-side XSS occurs when user input is passed into a server-side rendering engine without proper sanitization.
- Understood that different components of an application may sanitize input differently.
- Learned that some PDF engines support the file:// protocol, allowing access to local files on the server.
