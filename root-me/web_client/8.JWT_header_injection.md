#### Challenge Name
- JWT – Header Injection

#### Vulnerability Type
- The Keymaster now has his own site. He will only let you in if you give him the right secret and key.
- JWT Authentication Bypass: The server trusts an attacker-supplied public key from the JWT header for signature verification.
    
#### Goal
- Access the Matrix by forging a valid JWT as user Neo.

#### Initial Observation
- Recon identified `/key` and `/matrix` endpoints, both showing a login page
- Accessing `/matrix` redirects to `/key`
- Server: Werkzeug/2.2.2 Python/3.11.9
- A sample JWT was observed and decoded:
```
{"alg":"RS256","typ":"JWT"}  
{"user":"Trinity","message":"Wake up Neo, follow the white rabbit.","iat":1676637832}
 ```
- Application response:  
    “Sorry, you are not authorized to access the Matrix. Only the user 'Neo' can”
- Attempted `alg: none` attack, but it was rejected
- Identified that the application uses `RS256` (asymmetric signing)
    - Private key -> signing
    - Public key -> verification
- Checked for exposed public key endpoints, but none were found
- Considered whether the server accepts user-controlled keys via JWT header (`jwk`)
- Generated attacker-controlled key pair using OpenSSL:
`openssl genrsa -out private.pem 2048` 
`openssl rsa -in private.pem -pubout -out public.pem`
- Extracted modulus (`n`) and exponent (`e`) from the public key
- Understood that the server requires `n` and `e` to reconstruct the public key
#### Exploitation
- Converted the public key into JWK format using extracted `n` and `e`
- Created a malicious JWT header containing the attacker-controlled JWK
```{  
  "alg": "RS256",  
  "typ": "JWT",  
  "jwk": {  
    "kty": "RSA",  
    "n": "<attacker_modulus>",  
    "e": "AQAB"  
  }  
}
``` 
- Modified the payload:
```
{  
  "user": "Neo"  
}
```
- Signed the JWT using the attacker-controlled private key
- Constructed the token as: `base64(header).base64(payload).signature`
- Submitted the forged token in the `/key` input field
- The server used the public key from the JWT header (`jwk`) to verify the signature instead of a trusted key(stored key)
- Since the signature matched, the token was accepted
- Access to `/matrix` was granted

#### Server does:
- Takes the signature 
- Computes signature^e mod n -> recovers the hash 
- Independently hashes header.payload 
- Compares the two hashes 
- Match -> valid token

#### Remediation
- Never trust key material from JWT header
- Store trusted public keys server-side
- Verify against stored keys only
- Disable jwk/jku header parameters
- Use libraries that reject embedded keys by default

#### What I Learned
- RSA uses asymmetric cryptography: private key for signing, public key for verification
- If a server trusts user-controlled fields like `jwk`, attackers can control verification
- A valid signature is only meaningful when verified with a trusted key
- OpenSSL can be used to generate and manipulate keys for testing cryptographic attacks
