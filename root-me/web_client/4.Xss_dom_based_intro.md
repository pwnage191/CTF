#### Challenge Name
XSS DOM Based - Introduction

#### Goal  
Steal the administrator session cookie and use it to validate this chall.

#### Initial Observation  
- Opened the web page and explored how the application works.
- The application had two pages with input fields.
- The Main page accepts a number as input.
- The Contact page accepts a URL as input.
- In the main page, the input was stored inside a JavaScript variable without proper sanitization.

#### Approach  
- Checked where my input was placed in the page source.
- The input was inserted inside a JavaScript string.
- I closed the quote to maintain the syntax and injected JavaScript code to steal the cookie.
- When I tested the payload myself, it only returned my own cookie and did not validate the challenge.
- Then I visited the contact page which requires the URL to start with the challenge domain.
- I created a malicious URL with the payload and submitted it through the contact form.
- The server sends this URL to the admin bot for review.
- When the admin bot visited the link, the payload executed in the admin browser and sent the admin cookie to my request-capture server.

#### Solution
- Captured the administrator session cookie using Interactsh.
- Used that cookie value to validate the challenge.

#### What I Learned
- Learned how DOM-based XSS works when user input is inserted into JavaScript without sanitization.
- Understood that DOM XSS must execute in the victim's browser to steal sensitive data like cookies.
- Learned how to use tools like Interactsh to capture requests from the victim browser.
